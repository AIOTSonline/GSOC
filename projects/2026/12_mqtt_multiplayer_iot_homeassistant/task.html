Entry Task

Goal: Add an MQTT settings section to Catroid that can later be used by the MQTT bricks. This must be visible in the running app, persist correctly, and be covered by automated tests.

Requirements:
  1. Settings UI: Add a new “MQTT” entry in the app’s settings below (and similar to) the Raspberry Pi settings, with these fields:
    • MQTT extension (checkbox; default off)
    • Broker host (text; default e.g. 192.168.0.1)
    • Broker port (number; default 1883)
    • Use TLS (checkbox; default off)
    • Username (text; optional)
    • Password (text; optional; should not be shown in plain text while typing)
    • Client ID (text; optional)
  2. Persistence + accessor API: Values must be stored persistently (survive app restart). Provide a small, clean accessor API (Kotlin preferred) so future code can read: enabled, host, port, tlsEnabled, username, password, clientId. Keep naming clean and consistent (Clean Code).
  3. Basic validation: Port must be a valid integer (1–65535).
  4. Tests (automated): Add automated tests that verify at least:
    • defaults are returned correctly
    • values are persisted and reloaded correctly
    • enabling/disabling affects the “inactive” UI state OR at minimum the logic that determines whether fields are active
  5. Local coverage evidence (required): Provide a locally generated coverage report for the code you added/changed (HTML, screenshot, or zipped report), created for instance via Android Studio “Run with Coverage”. Include a short note stating how you generated it.
  6. Engineering style / process: TDD commit history is mandatory: tests first (red), minimal implementation (green), then refactors. Multiple small commits with meaningful messages, tests for following production code in separate commits; no squashing. Clean Code: excellent names, clear structure, minimal duplication. Avoid comments (few exceptions where they truly add value).
  7. Build & CI: The branch must compile and all tests must pass. Provide a GitHub Actions run link from your fork showing green build/tests, at least related to your code.
Submission format: Fork the repo, create a branch from develop, push your work, and provide links to your fork branch, compare link, GitHub Actions run, and local coverage artifact (zip/screenshot + how generated).

Required demo artifacts:
  • ~5 minute screen capture video walkthrough (showing the new MQTT settings UI, persistence, accessor API, tests, and explanations)
  • AI review artifact: Provide a link to a chat transcript/export where an AI tool reviews your submission and you address the key feedback.
Evaluation: Working feature in the app (settings + persistence), test quality, TDD, Clean Code, reliability, and clear demo.
